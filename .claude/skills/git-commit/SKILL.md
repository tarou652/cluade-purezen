# Git コミット支援

このスキルは、変更を**安全に・一貫した形式で**コミットするための手順です。
「確認してからコミット」が原則。勝手にコミット／プッシュしない。

## ワークフロー

必ずこの順で進める。

### 1. 変更を把握する

並行で実行して現状を掴む:

```bash
git status
git diff            # 未ステージの変更
git diff --staged   # ステージ済みの変更
git log --oneline -5  # 既存メッセージの語調・プレフィックス傾向に合わせる
```

- 何が・なぜ変わったのかを読み取る（差分の事実から。憶測で書かない）。
- 変更が**複数の無関係なまとまり**に見えたら、その旨をユーザーに伝え、分割コミットを提案する（最終判断はユーザー）。

### 2. ステージングを決める

- 関連する変更だけを `git add <path>` でステージする。`git add -A` は「全部入れて」と明示された時だけ。
- 生成物・秘密情報・無関係なファイルが混ざっていないか確認する。

### 3. メッセージ案を作る

**形式: `<prefix>: <日本語の要約>`**（1行目は50文字程度を目安）。

プレフィックスは変更の種類で選ぶ:

| prefix | 使う場面 |
|---|---|
| `feat` | 新機能・スライドや機能の追加 |
| `fix` | バグ・不具合の修正 |
| `docs` | ドキュメント・notes・コメントのみ |
| `style` | 整形・体裁（挙動を変えない） |
| `refactor` | 挙動を変えないリファクタ |
| `test` | テストの追加・修正 |
| `chore` | 依存更新・設定・雑務 |
| `build` / `ci` | ビルド設定・GitHub Actions など |

- 要約は**自分の言葉で「何をしたか」**を端的に。`docs: つまずきポイント集を追加` のように既存の語調に揃える。
- 補足が要るときは1行空けて本文に「なぜ」を書く。箇条書き可。
- **Co-Authored-By トレーラーは付けない**（このリポジトリの慣習に合わせる）。

### 4. 提示して確認を取る

ステージ内容（`git diff --staged --stat`）と**メッセージ案**をユーザーに見せ、「これでコミットしてよいか」確認する。修正要望があれば直す。

### 5. コミットする

確認が取れたらコミットする。複数行はヒアドキュメントで渡す:

```bash
git commit -m "$(cat <<'EOF'
docs: つまずきポイント集を追加

公開や Slidev でハマった点を後世向けに整理。
EOF
)"
```

コミット後、`git log --oneline -1` で結果を1行報告する。

## やらないこと（ガードレール）

- **プッシュしない**。`git push` はユーザーが明示的に頼んだ時だけ。
- フックを飛ばさない（`--no-verify`）／署名を無効化しない。フックが落ちたら原因を直す。
- `git reset --hard` などの破壊的操作をしない。
- ユーザーの確認なしにコミットを確定しない（「全部おまかせで」と言われた場合を除く）。
- メッセージは事実ベースで。差分にない内容を盛らない。

## 例

**Input**: deploy.yml と package.json を2デッキ対応に変更した状態で「コミットして」
**提示するメッセージ案**:
```
build: 2デッキ構成に対応（本=トップ / 使い方ガイド=/guide/）
```

**Input**: notes/summary.md だけ追記して「保存して」
**提示するメッセージ案**:
```
docs: 1・2章の読書メモを summary に追記
```
